# 09 — 提示词最佳实践（Prompting Best Practices）

> **版本**：v1.0 (2026-06-04)
> **用途**：帮助用户一次把任务交代清楚，减少反复补救
> **来源**：WorkBuddy 官方教程 — Prompts 章节

---

## 核心原则

Prompting 的重点不是写花哨提示词，而是像和同事协作一样，明确目标、约束、上下文和期望产出。

| 原则 | 说明 |
|------|------|
| **一次说清背景** | 开头交代到位，少反复补救 |
| **任务要具体** | 清晰描述目标、涉及范围与期望结果 |
| **说明约束** | 风格、对象、口径，哪些不要动 |
| **先分析再执行** | 复杂任务拆解后逐步实施 |

---

## 一个好的 Prompt 应包含什么

按以下顺序交代，效果最佳：

```
目标 → 资料 → 约束 → 产出 → 示例
```

### ① 目标（要做什么）
一两句说清目标与期望结果。
- ✅ "帮我写一份本周工作汇报，包含 3 个重点项目的进展和 1 个下周计划"
- ❌ "帮我写个汇报"

### ② 相关资料
文档、参考、示例输入输出。**主动 @引用是关键**。
- `@ima:项目进度表` → 引用具体资料
- `@tenant-docs:设计规范` → 引用知识库
- 直接粘贴关键数据/上下文

### ③ 限制条件
风格、结构、不要触碰的部分。
- "不要生成超过 500 字"
- "用中文，不要夹杂英文术语"
- "不要修改原始数据，只做汇总"

### ④ 完成的样子
内容、长度、格式、风格。明确"什么叫做好了"。
- "输出 Markdown 表格，3 列：项目名称、进展描述、风险状态"
- "用口语化风格，像跟同事聊天一样"
- "结尾附 1 句话总结"

### ⑤ 参考示例
范例/样稿，比文字描述更有效。
- 提供一个你满意的同类输出作为参照
- 说明"像这样就行，但 XY 换成 Z"

## 会话管理要点

| 场景 | 做法 | 原因 |
|------|------|------|
| 继续同一任务 | 留在原会话 | 延续上下文与状态 |
| 任务目标切换 | 新开一个会话 | 避免旧上下文干扰新任务 |
| 需要引用资料 | 主动 @ 文件/知识库 | WorkBuddy 不会自动看到所有内容 |

**上下文可见性规则**：
- ✅ **进入上下文**：引用的文件、连接器返回、匹配的记忆与技能
- ❌ **被排除**：无关历史、未引用资料、不相关的记忆

---

## 更有效的交代方式

### 1. 按顺序交代
```
目标 → 资料 → 约束 → 示例
```
这条顺序遵循 "认知负荷递减" 原则：先说最重要的，AI 越早理解意图效果越好。

### 2. 复杂问题拆成多轮
不要一次性塞入所有需求。先让 AI 做第一步，确认方向正确后再推进。

### 3. 结果偏了，补背景比反复重试更有效
如果 AI 理解错了，不要连续重试同样的指令。补一句背景解释（"我是做 A 领域的，这个数据是给 B 部门用的"），通常一次就能纠正。

### 4. 用示例驱动理解
给出一个"这样可以"的样本，比写 200 字的规范描述更精准。

## 新手常见误区

| 误区 | 问题 | 正确做法 |
|------|------|----------|
| 只给一句模糊命令 | "帮我看看""改一下" | 指明看什么、怎么改、什么算改好了 |
| 不说约束 | AI 自由发挥，结果不可控 | 明确格式、长度、语言、禁止项 |
| 塞太多无关内容 | 挤占 context window，核心信息被稀释 | 只给与任务直接相关的材料 |
| 换任务还沿用旧会话 | 旧上下文污染新任务理解 | 新任务 → 新会话 |

## Prompt 质量自检清单

写完后问自己：

- [ ] AI 看完知道要交付什么吗？（目标清晰）
- [ ] AI 知道有什么素材可用吗？（已 @ 资料）
- [ ] AI 知道不能做什么吗？（约束明确）
- [ ] AI 知道产出长什么样吗？（格式指定）
- [ ] 有没有给一个参考样例？（如果类型陌生）

**全部打勾 → 大概率一次过。**

## 进阶技巧

### 按任务类型选择交代策略

| 任务类型 | 重点投入 | 可简略 |
|---------|---------|--------|
| **数据分析** | 数据来源路径、口径说明 | 格式（AI 擅长挑合适形式） |
| **报告/文档** | 受众、用途、长度 | 具体措辞（让 AI 发挥） |
| **代码生成** | 技术栈、架构约束、测试要求 | 代码风格（AI 会匹配现有风格） |
| **调研/搜索** | 搜索方向、可信度要求、排除项 | 格式（报告结构可由 AI 确定） |
| **创意/设计** | 参考范例、避讳元素 | 技术细节 |

### 修复跑偏的产物

```
发现偏差 → 不等价重试 → 补一句背景 → 通常一次纠正

❌ 跑偏后连说 3 次 "不行，重来"
✅ "我重新说下背景：这个报告是给 CFO 看的，他关心成本不关心技术细节。基于这个重新生成。"
```
