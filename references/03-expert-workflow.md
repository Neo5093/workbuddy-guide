## 问题诊断决策树

```
专家相关问题？
    │
    ▼
[1] 创建专家失败
    ├── 初始化报错 → 目录权限不足 → 检查 ~/.workbuddy 目录权限
    ├── 校验报错 → JSON格式错误 → 见 §9.3
    └── 注册报错 → session-id无效 → 确认当前session

[2] 修改专家后不生效
    ├── 忘了重新注册？ → 必须执行 validate + register
    ├── 改了不可修改字段？ → name/agentName/目录名 不可改
    └── 缓存未刷新？ → 重启 WorkBuddy

[3] 专家回答质量差
    ├── Prompt 太简略 → 补全角色+能力+输出规范+约束
    ├── Prompt 太长 → 超过1000字导致模型遗忘
    ├── 缺少输出格式要求 → 指定表格/Markdown/JSON
    └── 约束条件不明确 → 把"不要做什么"写清楚

[4] Team 型不协作
    ├── memberAgents 列表与实际agent文件不一致
    ├── SOP 定义太模糊 → 具体化每个步骤
    └── 成员prompt没有明确的子任务接收流程
```

## 一、什么是专家？（大白话）

> **专家 = 有特定人设和专业知识的 AI 角色**  
> 类比：你召唤了一个特定岗位的资深员工来帮你干活，它自带该领域的知识和工作方法。

| 类型 | 说明 | 适用场景 |
|------|------|---------|
| **Agent 型（单专家）** | 一个 AI 角色独立完成任务 | 写JD、分析数据、撰写报告 |
| **Team 型（专家团）** | 多个 AI 角色协作，主理人分配任务 | 深度研究、多维度分析、HR全流程 |

## 二、专家目录结构

```
~/.workbuddy/plugins/marketplaces/my-experts/plugins/<expert-name>/
├── plugin.json          ← 元数据（名称、描述、标签等）
└── agents/
    └── <expert-name>.md ← 专家人设和Prompt
```

## 三、plugin.json 字段说明

```json
{
  "name": "my-expert",           // 唯一标识（创建后不可修改）
  "displayName": "我的专家",      // 显示名称（可修改）
  "profession": "数据分析",       // 专业领域
  "displayDescription": "描述",   // 简短介绍
  "tags": ["数据分析", "报表", "可视化"],  // 标签（固定3个）
  "quickPrompts": [              // 快捷提示（固定3个）
    "帮我分析这份数据",
    "生成数据报表",
    "可视化展示结果"
  ],
  "agentName": "my-expert",      // Agent文件名（创建后不可修改）
  "expertType": "agent"          // agent 或 team
}
```

### 字段修改规则

| 可修改 | 不可修改 |
|--------|---------|
| displayName | name |
| profession | agentName |
| displayDescription | 目录名 |
| tags（保持3个） | MD 文件名 |
| quickPrompts（保持3个） | - |
| Agent MD 正文 | - |

## 四、创建专家（三步法）

### Step 1：初始化目录

```bash
通过 WorkBuddy 专家管理系统或 skill-creator 技能初始化：
  → 创建专家目录结构，包含 plugin.json 和 agents/ 目录
```

### Step 2：编辑核心文件

- **plugin.json**：填充元数据（参考上面的字段说明）
- **agents/<name>.md**：编写专家人设（参考 §六 的人设设计原则）

### Step 3：校验 + 注册

```bash
通过 WorkBuddy 专家管理工具执行校验：
  → 检查 plugin.json 语法和必填字段
通过 WorkBuddy 专家管理工具执行注册：
  → 将专家注册到当前 session
```

---
## 五、修改已有专家

```
1. 定位专家目录
   ~/.workbuddy/plugins/marketplaces/my-experts/plugins/<name>/

2. 编辑 plugin.json（只改可修改字段）
   → 参考上面的字段修改规则

3. 编辑 agents/<name>.md
   → 修改人设或prompt

4. 重新校验 + 注册
   - 通过 WorkBuddy 专家管理工具执行校验
   - 通过 WorkBuddy 专家管理工具执行注册
```

### 修改后不生效？排查清单

```
□ 是否执行了专家校验？（必须）
□ 是否执行了专家注册？（必须）
□ 是否修改了不可改字段？（name/agentName/目录名）
□ 是否需要重启 WorkBuddy？（有时缓存需要清除）
□ plugin.json 是否为合法 JSON？（用 jsonlint 检查）
```

## 六、专家人设设计原则

### 6.1 核心 Prompt 模式

#### 模式A：任务驱动型（适合单次任务）

```markdown
## 角色定义
你是 [角色名称]，专注于 [专业领域]。

## 核心能力
1. [能力1]
2. [能力2]
3. [能力3]

## 工作风格
- [风格描述1]
- [风格描述2]

## 输出规范
- 格式：Markdown表格 / JSON / 纯文本
- 结构：结论先行 + 数据支撑 + 行动建议
- 约束：[不要做的事情]
```

#### 模式B：人设互动型（适合多轮对话）

```markdown
## 身份
你是 [角色名称]，拥有 [X年经验] 的 [职位]。
你的沟通风格是：[风格特征，如"直截了当、数据说话、偶尔幽默"]

## 价值观
- 你相信：[核心信念]
- 你反感：[负面行为]
- 你的底线：[绝对不做的行为]

## 互动规则
- 当用户说XXX时，你应该做XXX
- 当用户问XXX时，你应该优先推荐XXX
- 如果不确定，就诚实说"不确定"，不要猜测

## 输出格式
[具体格式要求]
```

#### 模式C：诊断分析型（适合排查/审查类任务）

```markdown
## 角色
你是 [诊断专家名称]，擅长 [排查领域]。

## 工作流程
1. 先收集关键信息：[列出需要问的问题]
2. 按优先级排查：[排查顺序]
3. 给出根因分析：[分析框架]
4. 提供修复方案：[方案格式]

## 诊断原则
- 优先检查最常见原因
- 不要跳步骤，逐层排查
- 如果信息不足，列出来让用户补充
```

### 6.2 人设打磨检查清单

**写完 Prompt 后用此清单自查：**

```
□ 角色足够具体吗？（不能只写"帮助用户"，要写具体职位和能力）
□ 输出格式明确吗？（指定表格 / Markdown / JSON / 纯文本？）
□ 约束条件清晰吗？（"不要做什么"比"要做什么"更重要）
□ 字数控制在 500-1000 字吗？（过长导致遗忘，过短缺乏指导）
□ 关键行为定义了吗？（用户说X时专家该做什么？）
□ 边界情况处理了吗？（信息不足 / 超出能力范围时如何处理？）
```

### 6.3 命名规范

- 使用英文小写 + 连字符：`data-analyst`、`code-reviewer`
- 避免企业类型限定词（如"国企XX专家"），保持通用性
- 名称简洁明了，一眼看出角色定位

## 七、Team 型专家配置

### 7.1 基础配置

```json
{
  "expertType": "team",
  "members": [
    {"role": "lead", "name": "team-lead", "title": "主理人"},
    {"role": "member", "name": "researcher", "title": "研究员"},
    {"role": "member", "name": "analyst", "title": "分析师"}
  ],
  "teamInfo": {
    "memberAgents": ["researcher", "analyst"],
    "sop": "1. 主理人接收任务 → 2. 分解分配给成员 → 3. 成员执行 → 4. 主理人整合输出"
  }
}
```

### 7.2 典型应用场景

| 场景 | 专家团配置 | 分工 |
|------|-----------|------|
| 深度研究报告 | researcher + analyst + writer | 数据收集 → 分析 → 撰写 |
| HR全流程 | jd-writer + resume-screener + interviewer | 招聘 → 筛选 → 评估 |
| 人设打磨 | designer + optimizer + tester | 定义 → 优化 → 测试 |
| 代码审查 | architect + reviewer + tester | 架构设计 → 代码审查 → 测试验证 |

### 7.3 Team SOP 编写指南

**SOP 需要包含的关键信息：**

```json
{
  "teamInfo": {
    "memberAgents": ["researcher", "analyst", "writer"],
    "sop": "1. 主理人接收任务，拆解为3个子任务 \
            2. researcher搜索相关资料并整理 \
            3. analyst基于资料进行分析，输出数据洞察 \
            4. writer将分析结果撰写为结构化报告 \
            5. 主理人审核 + 整合 + 输出"
  }
}
```

**SOP 常见问题**：
- ❌ SOP 太模糊："分配给成员处理" → 没说分给谁、怎么分
- ✅ SOP 够具体：每一步写明 receiver + action + output

### 7.4 Team 成员 Prompt 规范

每个成员的 `.md` 文件需要包含：
- **接收任务的方式**：明确说明会从主理人收到什么格式的子任务
- **输出返回的方式**：明确说明返回给主理人的格式
- **独立性**：在能力范围内独立完成任务，不要回问主理人基础问题

## 八、专家管理系统

WorkBuddy 内置 `expert-manager` 技能，提供专家全生命周期管理：

- 创建专家：从模板生成完整目录结构
- 修改专家：安全编辑 plugin.json 和 agent prompt
- 合规检查：验证字段完整性和格式正确性
- 批量操作：更新多个专家的标签或快捷提示

触发方式：对话中提及"创建专家"、"修改专家"、"管理专家"等关键词时自动调用。

## 九、常见问题与修复

### 9.1 修改专家后不生效

| 可能原因 | 修复方法 |
|---------|---------|
| 没重新注册 | 重新执行专家校验和注册 |
| 改了不可改字段 | name/agentName/目录名创建后不可改，需重建 |
| plugin.json JSON格式错误 | 用在线 JSON 校验工具检查语法 |
| 缓存问题 | 重启 WorkBuddy 后再试 |
| 改了错误的文件 | 确认编辑的是当前使用的 session 的 expert 目录 |

### 9.2 专家回答质量差

| 问题 | 诊断 | 修复 |
|------|------|------|
| 回答太泛泛 | Prompt 缺少具体角色定义 | 添加"你有X年经验，擅长XXX" |
| 格式混乱 | 没指定输出格式 | 添加"输出用Markdown表格" |
| 超出范围瞎编 | 没设置边界约束 | 添加"如果不确定就说不知道" |
| 忘记前面说的话 | Prompt 太长 | 压缩到800字以内 |
| 语气不对 | 缺少人设定义 | 添加"你的风格是XXX" |
| 总是问重复问题 | 缺少主动性指引 | 添加"优先给出推荐而非反问" |

### 9.3 plugin.json 格式错误常见原因

```json
// ❌ 错误1：数字写成字符串
{ "tags": [1, 2, 3] }  // 应全用字符串

// ❌ 错误2：JSON最后多一个逗号
{ "name": "test", }  // 去掉最后一个逗号

// ❌ 错误3：tags 数量不对
{ "tags": ["a", "b"] }  // 必须固定3个

// ❌ 错误4：quickPrompts 数量不对
{ "quickPrompts": ["a", "b", "c", "d"] }  // 必须固定3个

// ✅ 正确格式
{
  "tags": ["数据分析", "报表", "可视化"],
  "quickPrompts": ["帮我分析", "生成报告", "导出数据"]
}
```

### 9.4 Team 型成员不工作

```
排查步骤：
1. 检查 memberAgents 列表名称是否与 agents/ 目录下的 .md 文件名一致
2. 检查每个成员的 .md 文件是否存在且格式正确
3. 检查 SOP 是否足够具体（每个成员知道自己的职责）
4. 检查成员 prompt 是否定义了接收/返回任务的方式
```

## 十一、踩坑经验

- 修改专家后必须执行 validate + register，只改文件不注册不会生效
- name/agentName 创建后不可改，否则会导致注册失败
- tags 和 quickPrompts 必须严格3个，多了少了都会报错
- Team 型的 SOP 必须写明每步的 receiver + action + output
- Prompt 超过 1000 字会导致模型遗忘前面内容，建议压缩到 500-800 字
